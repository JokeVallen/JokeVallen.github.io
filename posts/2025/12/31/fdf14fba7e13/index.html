面向对象的三大特性

继承： 子类通过继承父类来获取基础特性，并且能够基于父类进行特异化开发，包括重用、修改和扩展父类行为。子类和父类又分别称为派生类和基类。基类的构造函数和终结器不会被派生类继承。继承具有传递性，例如 B 继承自 A ， C 继承自 B ，那么 C 就间接继承自 A ，这能够塑造继承关系的层次结构。在C#中，每个类仅允许继承一个父类。继承不仅能够提升代码的复用性也可以让程序的结构更加明晰。值得注意的是，派生类对基类成员的访问受限于基类中该成员的访问权限。

参考文档： 继承

在面向对象设计中，当多个功能或模块在本质上属于同一类事物，并且确实存在稳定且合理的 is-a 关系时，可以考虑将它们的共性逻辑和行为抽取到一个父类中，通过继承的方式实现代码复用和行为统一。这样可以避免重复实现相同的逻辑，并在父类中集中维护公共功能，从而提高代码的一致性和可维护性。父类可以通过访问修饰符对成员进行封装控制，只向子类暴露必要的扩展点，而隐藏其内部实现细节。与此同时，父类也可以通过抽象成员或虚方法对子类施加约束，明确哪些行为必须由子类提供具体实现。此类设计通常用于定义一套稳定的行为骨架，由父类负责整体流程或通用逻辑，而由子类负责差异化实现。在合理的设计前提下，通过新增子类来扩展系统行为，而不修改既有父类或调用方代码，可以在一定程度上符合“开闭原则”。需要注意的是，继承本身并不天然保证良好的扩展性，只有在继承层级清晰、父类职责稳定且抽象合理的情况下，继承才是一种有效的扩展手段；否则，过度或不恰当的继承反而会引入强耦合和维护成本。

封装： 将数据和行为封装为一个独立的单元，对外部隐藏具体的实现细节，仅提供必要的调用接口，从而简化外部调用和提升代码的复用性和可维护性。

参考文档： 面向对象编程 C#面向对象技术概述

在系统设计中，系统通常由多个功能模块组成。每个模块都可以被视为一个封装良好的独立单元，负责特定功能，同时通过明确定义的接口与其他模块交互。模块内部实现对外部模块隐藏，从而降低耦合，增强系统可维护性。通过这种封装，每个模块的内部逻辑可以独立维护，而模块对外暴露的接口可以被复用。在文档层面，可以分别为模块内部实现编写维护者文档，为对外接口编写调用文档（API文档），便于维护者理解模块内部实现，也便于调用者正确使用模块功能。

多态： 同一对象针对不同的的对象或应用情景而存在不同的行为，例如运行时的多态——重写，编译时的多态——重载，重写则是根据不同的实例对象而表现不同的行为，重载则是根据传递的参数而自动选择对应的行为。

参考文档： 多态

多态就像同一个按钮，不同对象按下时会有不同的表现，而你无需关心它们内部具体怎么实现。

值类型和引用类型

在C#中值类型继承自 System.ValueType 类，包括整型、浮点型、布尔类型、字符型四种内置的简单值类型，以及结构体和枚举两种复合值类型，这些简单值类型本质上就是结构体。引用类型继承自 System.Object 类，包括 class、delegate、array、object 等。

通常对于两个指向同一实例的值类型变量，它们之间是克隆副本的关系，所以各自的操作不会相互影响（除非变量由 ref、out、in 参数修饰）。对于指向同一实例的两个引用类型变量，它们都是存储对实例的引用，所以各自的操作会产生相互影响。

参考文档：值类型 引用类型

GC

GC（Garbage Collection，垃圾回收）是 .NET 运行时（CLR）提供的自动内存管理机制，用于管理托管堆（Managed Heap）中的对象生命周期。CLR 通过垃圾回收器负责对象内存的分配与回收，从而减少内存泄漏和悬垂引用等问题，但其目标并非“代替开发者管理所有资源”，而是自动管理托管内存。

.NET GC 的核心特征包括：基于可达性分析的标记-压缩算法、分代收集策略，以及对小对象堆（SOH）与大对象堆（LOH）的区别管理。

（1）基于可达性分析的标记-压缩算法
.NET GC 并非传统意义上的“标记-清理（Mark-Sweep）”算法，而是以 标记-压缩（Mark-Compact） 为主的回收策略。

其基本过程如下：

  • 标记阶段（Mark）
    GC 从一组称为 GC Roots 的根对象出发（包括线程栈上的引用、静态字段、CPU 寄存器中的引用以及 GC Handle 等），沿对象引用关系进行可达性分析。所有从根对象可达的对象都会被标记为“存活对象”。

  • 压缩与回收阶段（Compact & Reclaim）
    对于需要压缩的托管堆区域，GC 会移动存活对象，使其在内存中连续排列，并更新所有引用指向新的地址；不可达对象所占用的内存空间会被回收。通过压缩操作，可以有效减少内存碎片，提高后续内存分配的效率。

需要注意的是，是否进行压缩取决于堆的类型和运行时策略，并非所有堆区域在每次 GC 中都会被压缩。

（2）分代垃圾回收机制（Generational GC）
.NET GC 采用分代管理方式，其设计基于“弱代假说（Weak Generational Hypothesis）”，即：大多数对象的生命周期很短，只有少量对象会长期存活。

托管堆中的对象按其生命周期被划分为三代：

  • 第 0 代（Gen 0）：新创建的对象，回收频率最高

  • 第 1 代（Gen 1）：经历过至少一次 Gen 0 回收仍存活的对象

  • 第 2 代（Gen 2）：生命周期较长的对象

垃圾回收遵循以下原则：

  • Gen 0 回收仅扫描 Gen 0

  • Gen 1 回收会同时扫描 Gen 1 和 Gen 0

  • Gen 2 回收（即 Full GC）会扫描整个托管堆

对象在某一代垃圾回收中存活后，通常会被 晋升（Promotion） 到更高一代。由于高代对象被认为更稳定，代数越高，其垃圾回收触发频率越低，但回收成本越高。

（3）小对象堆（SOH）与大对象堆（LOH）
.NET GC 根据对象大小对托管堆进行进一步划分：

  • 小对象堆（SOH）：用于分配小于约 85,000 字节 的对象

  • 大对象堆（LOH）：用于分配大小 ≥ 85,000 字节 的对象

大对象具有以下特性：

  • 直接分配到 LOH，不经过 Gen 0 和 Gen 1

  • 仅在 Gen 2（Full GC）时参与垃圾回收

  • 出于性能考虑，LOH 默认不进行频繁压缩，以避免移动大对象带来的高开销
    （在较新的 .NET 版本中，LOH 支持通过配置进行按需压缩）

（4）重要补充说明

  • GC 只负责托管内存的回收，非托管资源（如文件句柄、数据库连接、显存等）仍需开发者显式释放，通常通过 IDisposable 模式实现。

  • 带有终结器（Finalizer）的对象在回收时会经历额外的生命周期阶段，可能会延迟内存释放。

  • 不同 GC 模式（如工作站 GC 与服务器 GC）在并发策略、暂停时间和吞吐量方面存在显著差异。

.NET 的垃圾回收机制是一套以可达性分析为基础、结合分代收集与内存压缩的自动内存管理系统。它显著降低了内存管理错误的发生概率，但并不意味着开发者可以忽略对象生命周期、资源释放成本以及 GC 行为对性能的影响。

参考文档：垃圾回收

反射

反射是一种在运行时检查并操作程序集元数据的机制。在 C# 中，可通过 System.Reflection 命名空间的 API 获取程序集、类型及其成员的信息，通常以 System.Type 为入口，在运行时动态构造对象、访问字段、属性或调用方法。由于反射绕过了编译期的类型绑定，相比直接调用通常存在一定的性能开销，因此应避免在高频或性能敏感的代码路径中使用，常见实践是在初始化阶段使用反射并对结果进行缓存。

C#中常见的反射操作对象包括Assembly、Module、ConstructorInfo、MethodInfo、FieldInfo、EventInfo、PropertyInfo、ParameterInfo、CustomAttributeData等。

参考文档：反射

StringBuilder和String

StringBuilder 顾名思义就是字符串构建器，对 StringBuilder 对象的操作是基于可变内存缓冲区的，这个特性减少了创建新对象的需求，相较于 String 类型，它更适合频繁对字符串进行改变的应用场景，然而它不是线程安全的。String 类型是不可变的，所以线程安全，对其进行的字符串增删改操作通常涉及到需要创建新的 String 类型对象。总而言之，对于需要频繁对字符串进行改变的情况建议使用 StringBuilder 类型，如果主要是读取或改变不频繁的情况那么建议使用 String 类型。

参考文档：StringBuilder String

常见集合类

（1）数组(Array)： 数组是一种长度固定的线性数据结构，在定义或初始化时必须明确指定其长度。数组支持通过索引直接读取和修改元素，但不支持动态增加或删除元素；若需要改变长度，必须创建新的数组实例并进行数据拷贝。在 .NET 的实现中，数组在内存中通常是连续存储的，因此按索引访问和修改元素的时间复杂度为 O(1)，非常适合存储数量固定、需要频繁随机访问的数据。

（2）列表(List<T>)： List<T> 是基于数组实现的泛型动态集合，可根据需要自动扩容，并且避免了 ArrayList 中由于使用 object 而产生的装箱与拆箱开销。在尾部添加元素（Add）时，其时间复杂度通常为 O(1)，涉及到扩容时就是 O(n)；在中间插入或删除元素（Insert / RemoveAt）时，需要移动后续元素，时间复杂度为 O(n)。因此，List<T> 适合以顺序访问和尾部操作为主的使用场景。

（3）栈(Stack<T>)： 栈是一种后进先出（LIFO）的数据结构，仅允许在栈顶进行元素的入栈（Push）和出栈（Pop）操作。其核心操作的时间复杂度均为 O(1)，但不支持随机访问，因此不适合需要频繁查询任意位置元素的场景。

（4）队列(Queue<T>)： 队列是一种先进先出（FIFO）的数据结构，元素从队尾入队（Enqueue），从队首出队（Dequeue）。其入队和出队操作的时间复杂度均为 O(1)，但同样不支持随机访问，适合用于任务调度、缓冲区等顺序处理场景。

（5）字典(Dictionary<K,V>)： Dictionary<TKey, TValue> 是一种基于哈希表实现的键值对集合，其中键必须唯一，而值可以重复。在哈希函数分布良好的情况下，查找、插入和删除操作的平均时间复杂度为 O(1)；在极端哈希冲突情况下，最坏时间复杂度可能退化为 O(n)。由于其高效的按键查找能力，Dictionary 非常适合需要快速映射和检索数据的场景。

（6）哈希集(HashSet<T>)： HashSet<T> 同样基于哈希表实现，但只存储单一元素，不允许重复值的存在。其查找、插入和删除操作在平均情况下的时间复杂度为 O(1)，非常适合用于去重、集合运算（如并集、交集、差集）等应用场景。

（7）哈希表(Hashtable)： Hashtable 是 .NET 中较早提供的基于哈希表的非泛型集合类型，键和值均以 object 形式存储，因此在使用值类型时会产生装箱和拆箱开销。在实现上，Hashtable 使用链地址法来处理哈希冲突，其设计思想与 Dictionary<TKey, TValue> 类似，但在类型安全性和性能方面均不如后者。因此，微软官方文档明确建议在新的项目中优先使用 Dictionary<TKey, TValue>，仅在需要与旧代码兼容时才考虑使用 Hashtable。

.NET提供了许多满足各类需求的集合类型，要求元素强类型的（泛型集合类型）、元素弱类型的（ Array 和 ArrayList 等将元素封装为 object 的集合类型）、多线程访问安全的（System.Collections.Concurrent）、集合不可变的（ System.Collections.Immutable ）以及监听集合变动的（ ObservableCollection<T> ）。

参考文档：集合

隐式继承和显式继承

在C#中通过英文符号 : 来实现显式继承，显式继承则需要显式指定基类，而隐式继承通常由编译器隐藏基类，例如声明自定义类时并不需要显式继承 System.Object ，而由编译器自动完成这个继承行为，所以声明的自定义类即使没有显式继承自 System.Object ，但实际上却是 System.Object 的派生类，在编译时就会有所体现，可以通过自定义类的实例查阅到可调用的方法中包括从 System.Object 继承而来的成员。

参考文档：继承

栈内存和堆内存

栈内存是按栈式分配规则使用的一段内存区域，主要用于保存函数调用过程中的执行上下文，如参数、局部变量、返回地址以及必要的寄存器保存信息。其分配与释放遵循严格的后进先出顺序，通常由编译器、运行时和硬件协同自动完成。

堆内存是用于动态内存分配的内存区域，其分配与释放不要求固定顺序，主要用于存储生命周期不受函数调用结构限制的数据，如对象和动态数组。堆内存由内存分配器或垃圾回收机制管理，并不等同于“堆”这一数据结构。

相比之下，栈内存分配和回收成本低、机制简单，但容量有限且生命周期受调用结构约束；堆内存容量更大、使用更灵活，但分配与管理成本更高，可能产生内存碎片或引入垃圾回收开销。

参考文档：基于栈的内存分配 托管堆

析构函数

析构函数使用语法 ~类名() 定义，无需访问修饰符、返回类型和参数。它实际上被编译器转换为重写 Finalize() 方法。

析构函数在垃圾回收器销毁对象时自动调用，但调用时机是不确定的，可能在对象不再被引用后的任意时间点执行。

官方强烈不建议在析构函数中释放托管资源，因为：

  1. 托管资源由垃圾回收器自动管理
  2. 析构函数执行顺序不确定，可能托管对象已被销毁
  3. 有析构函数的对象需要多代GC处理，影响性能

C#推荐使用 IDisposable 模式来管理资源，析构函数仅作为释放非托管资源的最后保障。

参考文档：终结器（析构函数）

抽象类和接口的区别

抽象类（Abstract Class）用于描述一类对象的共同本质，强调 is-a 。
它支持字段、构造函数以及已实现的方法，能够保存和复用状态，适合在继承体系中为子类提供统一的行为骨架和基础实现。抽象类只能被单继承，继承关系一旦确定，结构相对稳定，适合框架基类和模板方法模式。

接口（Interface）用于定义对象对外暴露的能力或行为约定，强调 can-do 。
它不维护实例状态，主要用于约束方法签名和交互协议，一个类可以实现多个接口，从而在不引入继承耦合的前提下获得多种能力。接口是实现模块解耦、依赖反转和多态扩展的核心工具。

核心区别总结如下：

  • 抽象类用于表达“是什么”的继承关系，接口用于表达 can-do 的能力关系

  • 抽象类支持状态和部分实现，接口侧重行为约定，不承担状态管理

  • 抽象类只能单继承，接口支持多实现

  • 抽象类更适合稳定的继承层级，接口更适合跨层级、可组合的设计

  • 抽象类偏向代码复用，接口偏向解耦和扩展

当需要共享状态和实现时使用抽象类；当需要定义能力、降低耦合、支持多实现时使用接口。
