随机-确定性边界模式（SDB Pattern）：LLM Agent进入生产系统的信任边界架构模式

TL;DR

SDB Pattern（Stochastic-Deterministic Boundary Pattern）是一种面向LLM Agent系统的架构设计模式。它的核心观点是：LLM负责理解目标、推理方案和生成行动计划，但不直接拥有改变确定性系统状态的最终权限。 任何进入确定性系统的行为，都必须经过一个由结构化意图、策略约束和权限验证组成的信任边界。SDB Pattern解决的不是”AI是否足够聪明”，而是：如何让一个具有推理能力但不可完全预测的软件主体安全地影响现实世界。

第一部分：Agent时代的新问题

1.1 从传统软件到Agent系统

传统软件系统：

系统假设：输入经过结构化约束，程序按照确定性规则完成决策和执行。

Agent系统：

系统发生了变化：输入不再只是数据，而是一个具有推理能力的软件主体提出的行动方案。这引入了一种传统软件系统中较少存在的问题，一个具备语义理解和行动规划能力，但行为结果具有随机性的非确定性主体，开始参与系统状态变化。

1.2 Intent-Execution Gap（意图-执行鸿沟）

Agent系统最大的风险，并不一定来自错误理解。更危险的是：AI理解了目标，并提出了一个逻辑合理、但系统不应该允许执行的行动。

示例：用户说”帮我降低云服务器成本。”Agent分析发现部分服务器利用率低，判断释放资源可以降低成本，调用 TerminateInstances(node-001, node-002)。结果生产核心节点被删除。

在这个案例中：AI没有幻觉、目标理解正确、推理过程合理。但问题在于：合理意图 ≠ 允许执行的行为。

这就是 Intent-Execution Gap（意图-执行鸿沟）。

传统业务系统通常通过结构化输入、接口契约和权限体系治理风险，而Agent系统额外引入了意图生成的不确定性。Agent系统需要额外解决：正确理解 + 合理推理 + 错误执行权限 = 系统事故。SDB Pattern针对的正是这个鸿沟。

第二部分：SDB Pattern核心原则

2.1 一句话定义

SDB Pattern：将LLM视为不可信随机输入源，而不是可信执行主体。这里的”不可信”并非指LLM没有价值，而是指其输出不能直接作为改变系统状态的授权依据。

2.2 信任模型变化

传统系统 SDB Pattern模型
输入来源 用户/API LLM Agent
主要风险 格式错误、非法参数 意图与权限不匹配
验证目标 保证数据正确 判断是否允许执行
执行主体 程序 确定性系统

核心变化：不是增加一个验证步骤，而是改变系统是否应该相信AI产生的行为请求。

2.3 核心概念关系

本文涉及两个核心概念：

概念 定义 角色
随机-确定性边界（S-D Boundary） AI输出与确定性系统之间的技术分界 技术形态
信任边界 AI行为进入系统前必须经过的验证关卡 意图/目的

两者关系：随机-确定性边界是信任边界的一种技术实现形式。 SDB Pattern通过建立”随机-确定性边界”这一技术层，来实现”信任边界”这个架构目标。信任边界是”为什么需要SDB Pattern”的答案，随机-确定性边界是”SDB Pattern怎么做到”的答案。

第三部分：SDB Pattern核心架构

整体数据流：

3.1 候选意图生成层（Candidate Intent）

职责：将LLM输出转换为结构化、可验证的行动描述。

说明：这一层不是为了消除AI的随机性，而是将自然语言中的模糊意图转换为系统可以审查的对象。

示例：用户说”删除测试账号player_test_001”，生成：

此时系统只获得一个候选行动，不执行。

3.2 约束验证层（Policy Boundary）

这是SDB Pattern真正的核心，执行四类验证：

① Schema约束：是否是合法操作？DeleteUser是否存在？

② 权限约束：Agent是否拥有该能力？注意：Agent权限不应等同于人类权限。推荐：

AI默认应该拥有更小权限集合。

③ 业务规则约束：例如禁止删除生产账号，允许删除测试环境账号。

④ 风险约束：根据影响决定执行路径：

风险等级 处理方式
低风险 自动执行
高风险 人工审批
临界风险 禁止自动执行

3.3 确定性执行层

通过验证后的对象（Verified Intent）才可以进入传统业务系统执行。执行系统不需要理解AI，它只接受已被批准的确定性命令。

第四部分：SDB Pattern与Tool Calling的关系

很多系统认为有Tool Calling就解决了Agent安全问题。这是错误的。

Tool Calling SDB Pattern
关注 调用格式 行为信任
问题 怎么调用 是否允许调用
对象 参数 意图
层级 通信协议 架构边界

关系：Tool Calling负责提供LLM与工具系统之间的调用协议，而SDB Pattern负责对这些调用请求进行身份、权限、策略和风险验证。二者不是替代关系，而是不同层面的问题。

第五部分：治理级别

SDB Pattern不是所有场景都需要最高安全等级。

级别 名称 适用场景 治理链
P0 基础治理 内部工具、非关键操作 Intent → Schema → Basic Policy → Execute
P1 安全治理 生产系统默认级别 P0 + Permission + Business Rule + Audit
P2 动态能力治理 AI可创建新能力的系统 P1 + Sandbox + Version Control + Capability Review
P3 高风险治理 金融、基础设施等 P2 + Shadow Validation + Human Approval + Controlled Rollout

第六部分：SDB Pattern解决什么，不解决什么

解决：

  • Agent权限失控：AI不能因为理解正确就直接执行。
  • 行为不可审计：所有AI行为转换为 Candidate Intent → Verified Intent，可追踪。
  • 随机性进入核心系统：把随机决策限制在确定性边界之外。

不解决：

  • AI幻觉：SDB Pattern不会让AI变聪明。
  • 纯内容生成：如文案、对话、创意生成，不存在执行边界。
  • 完全开放式自主智能：如果AI可以修改代码、创建服务、改变系统架构，需要更高层治理。

第七部分：核心结论

SDB Pattern的核心不是”如何让AI执行更多事情”，而是”如何防止AI因为执行能力增强而成为系统风险源”。SDB Pattern不是降低AI自主性，而是将自主性从”直接执行权限”转化为”受约束的行动能力”。

未来的软件系统可能出现一种新的基础原则：

LLM带来的最大变化，不是产生了更强的自动化工具，而是第一次出现一个能够理解人类目标、生成复杂行动方案、但其行为仍然具有随机性质的软件主体。

因此：AI Agent进入生产系统面临的关键问题之一，不只是能力提升，而是如何建立与其能力匹配的信任边界。 SDB Pattern试图定义的，就是这条边界。
