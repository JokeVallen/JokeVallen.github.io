Intent-Controlled Routing Pattern（ICR）

一种通过中间语言与映射规则治理LLM调度能力的架构模式

TL;DR

ICR（Intent-Controlled Routing Pattern）是一种面向LLM Agent系统的调度治理架构模式。它的核心思想是：LLM不直接调用业务系统，而是将自然语言意图翻译为一种结构化的中间语言（Intermediate Language, IL），再由开发者自定义的映射规则将IL转换为确定的业务系统指令。 开发者通过定义”IL→指令”的映射规则来控制LLM能够调度哪些系统能力。ICR解决的不是”LLM是否可信”，而是”如何让开发者对LLM的调度能力拥有完整的控制权”。

第一部分：问题的本质

1.1 无限语言与有限系统之间的鸿沟

LLM Agent进入生产系统的常见架构：

这个架构隐含一个假设：LLM理解需求后，可以直接决定调用业务系统的什么能力。

如果你的业务系统拥有明确的、可枚举的能力集合：

问题在于：如何确保LLM只调度它应该调度的能力？

这本质上是无限的自然语言表达空间与有限的确定性业务系统能力之间的映射问题。LLM可以理解无数种表达方式，但业务系统只有有限的、确定的API。如何在两者之间建立一座可控的桥梁，是ICR要解决的根本问题。

1.2 更接近开发者直觉的解决方案

开发者的直觉通常是这样的：

“我的系统有这些能力，我允许LLM通过某种方式调用它们，但调用方式必须由我定义。”

如果用自然语言描述这个需求：

“用户说一句话，LLM理解它，翻译成一种我能控制的语言，然后我定义这个语言如何变成系统调用。”

这就是ICR的核心思想。

1.3 ICR与现有架构的关系：不是重叠，而是演进

一个常见的疑问是：ICR是否与API网关、语义路由等现有架构重叠？

与API网关的区别：API网关基于请求的元数据（URL、Header）进行路由和策略实施，位于”业务系统”一侧。ICR的输入是LLM生成的结构化意图（IL），它位于”LLM”与”业务系统”之间，决策依据是意图本身包含的语义信息（如”要做什么”、”作用在什么对象上”），而非网络层面的元数据。

与语义路由的区别：语义路由通过LLM理解用户意图，将请求分发到最合适的子Agent或处理流程。ICR在此之上增加了一个**”映射规则层”——语义路由的”路由”是AI的动态决策结果**，而ICR的”路由”是一个可编程的过程。开发者通过预定义的规则，明确规定”意图A”在什么条件下可以映射为”能力B”，什么情况下会被拒绝或需要审批。

本质差异：

对比维度 API网关 语义路由 ICR
核心输入 HTTP请求 自然语言用户输入 结构化意图（IL）
路由依据 网络/协议元数据 LLM对语义的理解 开发者预定义的映射规则
控制权归属 平台/运维团队 AI模型 开发者（通过规则）

ICR可以看作是建立在语义路由之上的一层”设计时保险“——它不是多此一举，而是为LLM Agent的”路由”行为提供了生产环境所必需的可控性、可审计性和确定性。

2.1 整体数据流

2.2 三层详解

① LLM翻译层

职责：将自然语言需求翻译为结构化的中间语言（IL）。

核心约束：LLM只负责”翻译”，不负责”决定”。翻译的结果是IL——一套抽象的、与具体业务系统无关的表达。

IL示例：

关于IL的格式约束：ICR要求IL具备”结构化、确定性、可验证”三个特征，但并不强制要求IL必须使用JSON Schema。JSON Schema是当前最主流的实现方式——它在LLM结构化生成、生态成熟度和人类可读性方面都有明显优势，ICR将其视为推荐选项而非强制标准。开发者也可以选择Protobuf、Avro，或自定义DSL，只要满足”结构化、确定性、可验证”三个条件即可。

LLM如何将意图翻译为指定的IL？ 这是ICR可行性的关键。当前技术已通过**结构化生成（Structured Generation）**解决了这个问题。具体方式包括：

  • JSON Schema硬约束：主流LLM服务商（OpenAI、Anthropic）和开源生态（NVIDIA NIM）均已支持guided_json、response_format等参数，能够在Token生成层面物理屏蔽不符合格式的输出，LLM从物理层面无法生成枚举列表之外的词。
  • 上下文学习：在Prompt中提供业务系统字典和翻译示例（Few-shot），LLM可以基于开发者提供的”能力清单”进行翻译——它是一个选词填空的过程，而非创造的过程。

ICR的设计承认了LLM的翻译可能出错这一事实，因此不要求100%的翻译准确率。翻译层之后还有映射规则层做最终审核——格式错误或语义偏离的IL会被拦截，不会流向业务系统。

② 映射规则层

职责：开发者定义的规则，将IL映射到业务系统指令。

这是ICR的核心——它不是一个”验证层”，而是一个”控制层”。

映射规则示例：

映射规则可以：

  • 批准映射：IL直接转换为业务指令
  • 拒绝映射：该IL不产生任何业务指令
  • 条件映射：根据上下文决定是否映射
  • 部分映射：只有部分IL字段被映射
  • 延迟映射：进入人工流程

映射规则的具体实现方式：

实现方式 说明 适用场景
静态配置（YAML/JSON） 声明式规则定义，集中管理 大多数场景，规则清晰明确
代码式（策略模式） 动态路由逻辑，支持复杂计算 需要依赖外部状态或动态判断的场景
可视化低代码配置 决策表、规则流等图形界面 业务人员自助配置规则

规则引擎与执行引擎的分工：ICR的映射层可以进一步拆分为两个独立的子层——规则引擎负责”决策”（判定一个IL该做什么），执行引擎负责”执行”（将决策落地为具体的调用序列）。

规则引擎 执行引擎
核心问题 “这个意图应该做什么？” “如何安全、可靠地完成这个操作？”
输入 IL 规则引擎输出的”执行计划”
输出 具体的”执行计划” 实际的业务系统API调用结果
核心能力 条件匹配、决策路由、策略判断 指令链编排、参数映射、事务管理
开发者视角 业务决策逻辑 技术实现细节

为什么需要分离？ 分离的好处在于：修改规则不影响执行引擎，反之亦然；规则引擎可以独立测试匹配逻辑，执行引擎可以独立测试编排能力；规则引擎可以从YAML换成数据库驱动，执行引擎可以从同步换成异步——分离后替换其中一个不影响另一个。

职责：执行业务系统指令，返回结果。

执行层不关心IL从何而来——它只执行”已被映射层确认”的指令。

2.3 核心设计哲学

ICR与”验证/审核”模式的根本区别在于：

验证/审核模式 ICR（控制映射模式）
控制时机 事后（LLM生成后检查） 事前（设计映射规则）
控制方式 检查是否允许执行 定义什么可以映射
开发者角色 配置权限策略 定义转换规则
LLM自由度 可以生成任意请求，由系统拦截 只能生成IL，由规则决定是否映射
思维模型 “防御” “控制”

ICR的核心是将控制权从运行时前置到设计时——开发者通过定义映射规则，在架构层面限定了LLM能调度什么能力。

为什么不在Prompt里直接写规则？ 这是一个常见的疑问。Prompt是”建议”而ICR是”强制”：

维度 仅用Prompt约束 ICR（Prompt + 映射规则）
约束性质 建议性 强制性
绕过难度 通过Prompt注入即可绕过 系统级，不可绕过
确定性 概率性遵守 确定性执行
可审计性 无法审计AI是否遵守了规则 所有意图→指令的映射均有日志
变更成本 每次规则变更需修改Prompt、测试、上线 仅修改映射规则配置，无需动LLM

Prompt负责引导翻译质量，ICR负责强制安全边界。两者分工不同，缺一不可。

第三部分：渐进式架构——开箱即用与深度定制

ICR的一大优势是支持渐进式复杂性，允许不同层次的用户选择适合自己的使用方式：

  • 对普通开发者（开箱即用） ：提供业界成熟、通用的IL标准（如OpenAPI Schema、GraphQL、JSON-RPC）。不需要重新发明轮子，直接享受”自然语言→标准IL→业务系统”的解耦红利。
  • 对高级/特定领域开发者（深度可扩展） ：允许自定义IL语法与解析规则。架构层提供了这种”插拔能力”，把系统的决定权重新交还给了开发者。

第四部分：指令链编排与能力组合

ICR不仅支持IL到单个指令的一对一映射，更支持将IL映射为预设的”指令链”：

工程价值：

  • 原子性保证：将多步业务操作封装为原子单元，避免中间状态失败。
  • LLM不需要知道实现细节：只需触发意图，无需了解背后需要的5个步骤。
  • 事务性与可观测性：指令链的执行可被完整追踪和回滚。

指令结构的灵活性：ICR对”IL→指令空间”的映射不强制约束指令结构。映射输出可以是单个原子API调用、线性有序的指令链、甚至包含并行分支和条件分支的DAG——开发者可以根据业务需求自由定义。这种灵活性使得ICR既能适配简单的查询操作，也能支撑复杂的跨系统业务流程。

第五部分：通过指令集粒度调节AI自由度

开发者通过设计指令集的抽象粒度，动态调节给AI的灵活性：

指令粒度 控制方式 适用场景
细粒度指令集 暴露原子指令，允许LLM自由组合 高智商Agent、探针类工具
粗粒度指令集 只暴露宏指令，LLM只能做”多选一” 高风险、强合规场景
开放程度 控制层级 适用场景
高灵活性 LLM可以自由组合细粒度指令 探索性任务、数据分析
中灵活性 LLM在预定义模板范围内编排 标准业务流程
低灵活性 LLM只能触发预设的指令链 高风险操作、合规场景

第六部分：ICR的技术可行性

ICR的每个核心模块都有成熟的技术支撑，不存在底层技术瓶颈：

模块 技术可行性 说明
结构化生成（翻译层） ✅ 极高 NVIDIA NIM、OpenAI、Anthropic等均有生产级实现，支持guided_json、response_format等硬约束方式
规则引擎（映射层） ✅ 极高 Drools（复杂推理）、QLExpress（轻量AI友好）、Easy Rules（简单场景）等方案成熟
执行引擎（编排层） ✅ 极高 Camunda（BPMN标准）、Temporal（高容错分布式编排）、Airflow（数据管道）等均可复用

执行引擎如何对接业务系统？ 执行引擎可以通过API适配层将IL映射为业务系统的标准REST API调用；通过Activity机制（如Temporal）确保调用的可靠性和重试能力；通过信号机制处理需要等待外部反馈的长时间操作。业务系统本身不需要为适配ICR做任何改造，它只接收标准化的确定性指令。

ICR的价值不在于发明新技术，而在于把这些成熟技术组合成一个面向LLM Agent的治理框架。

第七部分：ICR解决什么，不解决什么

解决：

1. LLM调度能力范围控制

开发者通过映射规则定义LLM能调度什么——不是运行时拦截，而是设计时限定。

2. 业务系统与LLM的解耦

业务系统不需要为LLM做任何改造——它只提供标准指令集，映射规则负责翻译。

3. 意图与执行的清晰分离

自然语言→IL→业务指令，每一层职责清晰，可独立演进。

4. 可审计性

所有IL→指令的映射记录可追溯，映射规则本身也是系统的一部分。

5. 防止防御性代码在业务系统中蔓延

所有安全约束、规则过滤、能力遮罩全部在IL映射层完成。业务系统保持”无知”与纯粹——只暴露确定的、原子化的指令集，不关心请求来自人类、程序还是LLM。业务代码不包含任何AI相关的防御逻辑。

不解决：

1. LLM的理解错误

如果LLM把”删除测试账号”翻译为”删除所有账号”——ICR不会纠正这个错误，只会按规则映射（或不映射）。

2. 业务逻辑的正确性

ICR只控制”是否允许映射”，不保证映射后的业务操作在语义上是”正确”的。

3. 纯内容生成场景

对话、创意写作等不存在”业务指令”的场景，ICR不适用。

4. 完全无边界自主AI

对于没有固定业务指令集、完全开放式探索的实验型自主AI，ICR无法构建映射体系——它的适用前提是业务系统拥有可枚举的、确定性的能力集合。

第八部分：治理级别

ICR可根据场景选择不同治理强度：

级别 名称 适用场景 映射规则强度
L0 直接映射 内部原型、低风险工具 所有IL自动映射到对应业务指令
L1 条件映射 一般业务系统 根据环境、参数等条件决定是否映射
L2 严格映射 生产环境、关键业务 仅预定义的白名单IL可映射，其余拒绝
L3 审批映射 金融、基础设施等高风险场景 关键操作需二次确认后才映射

第九部分：理论闭环——行为空间的确定性收敛

ICR在数学语义层面为LLM的”行为空间”画出了一个确定性的闭包：

  • LLM拥有无限的”语义理解自由度” ：支持各种刁钻、模糊的自然语言输入。
  • 开发者拥有绝对的”系统调度控制权” ：最终能落地的操作永远不超纲。

不管LLM怎么翻译自然语言，最后都只能映射到指定的指令集合中。

第十部分：ICR的适用AI类型

ICR的适用性取决于一个核心变量：AI是否具备对业务系统产生”状态变更”的能力，以及其能力空间是否可预定义。

AI类型 ICR是否适用 说明
非自主Agent（单轮工具调用） ✅ 适用 能力边界清晰，映射规则可直接覆盖
受限自主Agent（多轮、预定义工具集） ✅ 适用 需要为多轮循环中的每一步都配置映射规则
自主Agent（动态生成工具） ⚠️ 需人工补充规则 动态生成的新工具需要人工注册映射规则后才能被调用
RAG Agent（只检索不操作） ❌ 不适用 不涉及系统状态变更，没有”执行”这个动作
纯内容生成 ❌ 不适用 不存在”业务指令”的映射对象

ICR对自主性的限制：ICR的设计假设是”能力边界是预定义的、静态的”。对于能够动态生成新工具的自主AI，ICR要求新增能力必须人工注册映射规则后才能被调用——这从根源上避免了AI自主生成后门工具绕过管控。这种”受限自主”的设计哲学，使得ICR在生产环境中既能保留AI的灵活性，又能确保系统安全。

第十一部分：与SDB的关系

ICR与同期整理的SDB（Stochastic-Deterministic Boundary，随机-确定性边界模式）共享同一核心前提——LLM不应直接调用业务系统。两者差异在于：

ICR SDB
核心问题 LLM的调度能力由谁控制 AI输出是否值得信任
控制方式 映射规则（事前定义） 约束验证（运行时检查）
思维模型 “开发者控制LLM能调度什么” “系统验证AI是否被允许执行”
开发者角色 定义”IL→指令”的映射表 配置验证规则和权限策略
适用场景 需要精细控制LLM能力边界的系统 需要审计和合规的安全敏感系统
关键词 调度控制、映射规则、中间语言 信任边界、约束验证、安全检查

ICR关注”控制” ：如何通过映射规则让开发者定义LLM能调度什么。这是一种设计时的控制思维。

SDB关注”防御” ：如何通过验证边界确保LLM的输出在运行时安全。这是一种运行时的防御思维。

两者可以独立使用，也可以组合：ICR负责定义能力边界，SDB负责加固信任边界。组合使用时，ICR在前（事前收敛行为空间），SDB在后（运行时二次安全校验），形成”设计时+运行时”的双重防护。

第十三部分：结语

ICR的核心思想来自一个朴素的问题：

当LLM成为系统入口时，开发者如何确保它只调用应该调用的能力？

答案是：把控制权留给开发者。

ICR通过”中间语言→映射规则→业务指令”的三层结构，在设计阶段就定义了LLM的能力边界。

三个核心设计原则：

  1. 把灵活性留给用户——LLM理解各种自然语言表达。
  2. 把控制权留给开发者——通过映射规则决定LLM能调度什么。
  3. 把纯洁性留给业务系统——业务代码不包含任何AI相关的防御逻辑。

开发者不需要理解LLM的内部机制，只需要定义一套映射规则——什么样的IL可以映射到什么样的业务指令。LLM负责翻译，映射规则负责控制，业务系统负责执行。

职责清晰，边界明确，控制权在开发者手中。

ICR的长期价值：ICR所代表的”意图治理层”是否会成为未来AI软件架构的基础组件，取决于一个前提——AI是否会大规模进入生产系统并承担执行责任。如果这个前提成立，那么”如何让AI在可控范围内影响系统状态”就会成为一个结构性问题，而ICR提供的正是这个问题的参考答案。

这也许不是最”智能”的解决方案，但它是开发者最熟悉的解决方案——就像数据库查询、API路由、配置管理一样，通过明确的规则来控制系统的行为。它可能在未来的某一天被更优雅的模式取代，但在当前，它为”AI如何安全地接入业务系统”提供了一个具备工程可行性的具体路径。
