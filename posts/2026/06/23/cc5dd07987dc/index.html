Unity计时工具GlobalTimer

更新于2026-06-24|开源工具
|浏览量:

前言

在游戏开发中计时任务的需求是很常见的，通常我们可以通过协程、Unity提供的刷新回调、C#计时器来提供计时任务，这些在本工具库中被视为时间源，不同的时间源可能存在不同的优势和限制，所以我们决定设计一个工具库来统一管理它们，对于业务开发者而言，不希望每次计时任务都重复造轮子，他们只需要理解各个时间源的适用时机并为此使用它们即可，所以我们对此进行了封装。

功能特点

多种时间源

时间源是指不同的计时实现策略，但它们本质都是提供计时支持，但是各有优劣和适用场景。我们提供了两个概念 原子级增量计算方式 和 原子级驱动调度时机，二者从两个维度上进行组合，扩展不同类型的时间源，覆盖多种应用场景，包括但不限于 Unity运行时、编辑器工具、服务端、后台挂机、自定义循环。

计时任务组

有时候多个计时任务可能在业务逻辑上是相关联的，分组机制使得可以以组为单位对计时任务进行批量操作。

结构体句柄

我们没有采用传统句柄模式，将计时任务实例封装，我们选择通过世代编号机制将句柄封装为一个代表一次计时任务的身份标识，然后通过内部提供的相关方法以及句柄的静态扩展方法，间接实现了与传统句柄一样的便捷调用，但基于此我们避免了悬挂引用的问题，也避免了句柄带来的堆分配。

限制

同时存活的计时任务数量上限

目前该上限为 2048，意味着存活的计时任务数不可超过该上限，因为内部使用的数组，并且没有提供自动扩容机制，这是一个远超正常业务需求的设定，但我们需要说明，过多存活的计时任务本身就是不推荐的用法，但工具库不能基于常规业务场景设定，如果确实存在大量计时任务需要同时执行，可以考虑对它们进行排队分批执行或者对任务进行嵌套组合，工具库牺牲自动扩容是为性能让路。

仅支持运行时

1.0.1-beta 及之前的版本不支持编辑器非运行时，但后续版本可能会提供支持。

性能

指标 结论
GC 分配 所有运行时操作均为 0 字节分配，无 GC 压力
注册/取消 亚毫秒级（< 1 ms），适合高频动态创建
批量清理 1000 个计时器仅需 ~2 ms，场景切换无感知
分组操作 CancelGroup 极快（< 0.3 ms），PauseGroup 适中（500 个约 7 ms）
查询 API 10000 次调用 < 0.5 ms，可放心在 Update 中高频使用
容量压力 2048 容量安全边界充足，超额时优雅降级
