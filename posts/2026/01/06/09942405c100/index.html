AssetBundle（一）

|浏览量:

术语表

术语 阐述
直接依赖 如果为依赖划分层级，直接依赖就是第一层级。指资源或代码直接引用的对象，如 Prefab 直接引用的材质。
间接依赖 如果为依赖划分层级，间接依赖就是除第一层级外的其它层级。如材质引用的纹理，对 Prefab 而言即间接依赖。
完全依赖 直接依赖 + 间接依赖
调试版本 用以检验程序逻辑和挖掘程序漏洞的版本。通常包含调试符号、未压缩、日志全开。
发行版本 功能基本稳定，已知漏洞较少，能够比较稳定投入实际使用的版本。通常经过优化、压缩、移除调试信息。
开发环境 进行程序开发的环境。包含编辑器、调试工具、测试设备等。
生产环境 程序正式投入使用的环境。即玩家/用户实际运行的环境。
编辑器时 项目在完全打包成为可执行程序前开发者进行开发的时段。包含资源构建、代码编译等阶段。
运行时 项目打包为可执行程序并运行的时段。即程序在用户设备上执行的阶段。
主清单 构建时生成的与目录同名的 AssetBundle，内含 AssetBundleManifest 对象，用于查询所有 Bundle 的信息及依赖关系。
依赖加载 在加载目标 AssetBundle 前，先递归加载其所有依赖 Bundle 的过程。
冗余资源 同一份资源因被多个 Bundle 引用而被重复打包的现象。
CRC/Hash 用于校验 AssetBundle 完整性的校验值，常用于版本比对与增量更新。

什么是 AssetBundle？

AssetBundle 是一个存档文件，包含可在运行时由 Unity 加载的特定于平台的非代码资源（比如模型、纹理、预制件、音频剪辑甚至整个场景），是 Unity 中实现资源与代码分离、动态分发与按需加载的基础设施。AssetBundle 彼此之间可以存在依赖关系。例如，一个 AssetBundle 中的材质可以引用另一个 AssetBundle 中的纹理。为了提高通过网络传输的效率，可以根据需求选用内置算法（LZMA 或 LZ4）来压缩 AssetBundle。AssetBundle 需要通过 AssetBundle.LoadFromFile、UnityWebRequestAssetBundle 等 API 加载，并且加载时 Unity 会自动处理其内部序列化数据和资源数据的解析。直观一点，我们可以把 AssetBundle 理解成资源文件特定于 Unity 项目的压缩包，就像 .zip .7z 等文件，当然这仅是它的压缩方式 LZMA 时的类比，它也可以选择不压缩或者以 LZ4 压缩，但无论哪种方式都可以将 AssetBundle 理解为对资源的封装。

AssetBundle 的内容蓝图

  • AssetBundle可以划分为两个数据区域：

    • 序列化数据区域：存储对象的序列化数据。序列化数据区域偏向于 "框架"，这些通常是我们的游戏程序中通过 Unity 引擎构建的的部分，比如场景（包括各种游戏对象和组件等）和预制体（包括子对象和各种组件等）等，那么这些表示的是当前包体中的内容组织框架，不与具体的资源存储在一起，因为资源可替换，但它通常意味着对资源存在依赖。

    • 资源数据区域：存储诸如纹理、音频等程序主体依赖的资源。资源数据区域偏向于 "内容"，它们通常需要转换为单独的二进制块，避免与序列化数据区域强绑定能够让大体量资源在单独的线程上加载从而避免阻塞游戏主线程，内容意味着可替换，比如相同的游戏对象可以替换不同的纹理资源。

  • .manifest文件：用于开发者查看或调试的清单文件，不应用于发布版本的生产环境运行时。

  • 用于运行时查询的主清单文件：在构建 AssetBundle 后会有一个单独的 AssetBundle 用于存储 AssetBundleManifest 这个资源清单文件，通过这个清单文件，我们可以快速索引各个 AssetBundle 直接或间接依赖的其它 AssetBundle 的名称。所在 AssetBundle 通常与构建目录同名。

AssetBundle 可以用来做什么？

  • 项目资源的组织、压缩、解压、加载和卸载等；
  • 资源分包，减少初次安装包体积；
  • 分批加载游戏非代码资源，提高游戏启动速度；
  • 提供特定于平台或画面级别的资源；
  • 游戏资源的热更新；
  • 游戏 DLC；
  • 让游戏资源以一种更通用更兼容 Unity 引擎的形式存储；
