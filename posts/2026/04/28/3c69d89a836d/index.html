前言

代码生成是项目开发中常见的需求之一，例如通过 "Attribute + 编译期代码生成" 可以帮助开发者避免编写许多重复性的注册代码，例如你需要在项目中为多个单例进行统一初始化，首先你需要先收集这些单例类型，其次通过指定逻辑进行初始化，传统方式则是每新增加一种单例就在对应脚本集合初始化器中添加一行，或者添加一行初始化代码，无论是哪种都是带有侵入性的，且一旦数量过多就会变得难以维护，进而导致项目异常概率提升，代码生成的思路则是基于代码逻辑动态完成重复性代码生成，这样每次修改只需要重新进行生成即可，而不需要去修改原有代码逻辑，也减轻了开发者的负担。本文将基于一个轻量级的代码生成框架，结合 Unity API 和 C# 设计一个代码生成的应用示例，借此展示代码生成在开发工作中的重要作用。

说明

本节详细解释如何利用 CodeGenerator 框架构建一个自动化过滤器注册系统，彻底消除手动维护注册代码的繁琐工作。

1. 解决什么问题？

在 Unity 编辑器工具开发中，经常会遇到“过滤器”模式：定义一系列实现了 IFilter<TSource, TInput> 的类，然后需要一个中央工具类来查找、创建和管理这些过滤器。传统做法是每新增一个过滤器，都要在工具类的注册列表里手动添加一条代码，不仅容易遗漏，而且随着过滤器数量膨胀，维护成本激增。
代码生成的思路是：仍然自由添加过滤器类，但注册代码完全由生成器扫描并自动生成，从而保持“新增类 → 执行生成 → 自动完成注册”的闭环。

2. 核心标记与约定

  • [FilterAttribute]：标记一个类是过滤器，可选的 DisplayName 用于 UI 展示或调试。
  • [TargetAttribute]：指定该过滤器关联的“目标类型”。例如，某个过滤器只针对特定组件或数据起作用，通过此特性声明关联，允许多个 [Target] 堆叠。
  • IFilter<TSource, TInput>：通用的过滤器接口（未在片段中展示，但为约定），包含 bool Filter(TSource source, TInput input) 方法。

约定确保了生成器能够通过统一的方式获取每个过滤器的泛型参数，并生成类型安全的代码。

3. 模板与生成流程

所有动态部分由模板文件 FilterUtility.cs.template 定义，其中三个占位符被定义在 $[PLACEHOLDER] 中：

占位符 生成内容
$[FILTER_REGISTRY] 所有过滤器的类型注册表，每一项为 (sourceType, inputType, filterType) 元组。
$[FILTER_TARGETS] 每个过滤器类型与其目标类型列表的映射字典。
$[FILTER_DISPLAY_NAMES] 过滤器类型到显示名称的映射。

执行 Generate FilterUtility 菜单项时，GeneratorMediator 调用 FilterUtilityGenerator.Generate(string template)，后者通过正则表达式替换上述占位符。替换逻辑利用 Unity 的 TypeCache （仅编辑器可用）高效获取所有标注了 [FilterAttribute] 和 [TargetAttribute] 的类型，然后用 StringBuilder 拼装出 C# 集合初始器代码。

5. 框架的复用与扩展

本示例是 CodeGenerator 框架的一个具体应用，其核心价值在于将可变部分（注册代码）交给生成器，而框架本身提供模板管理、中介者调度和实用工具。如果你想为其他场景（如验证器、处理器、事件系统）生成类似的工具类，只需：

  1. 派生新的 IGenerator<string, string> 并标记 [GeneratorConfig]；
  2. 编写对应的 .template 模板；
  3. 在 CodeGenerator 入口类中添加菜单项调用即可。

由于 GeneratorMediator 会通过 TypeCache 自动扫描所有标记了 [GeneratorConfig] 的生成器，新增一个生成器几乎不需要修改现有调度逻辑，极大地提升了扩展性。

6. 辅助工具组件

  • GenerateHelper：将 System.Type 转换为 C# 友好的类型名，支持泛型、可空类型、数组等，保证生成的代码语法正确。
  • GenerateUtility：封装了模板读取、代码写入、自动刷新 AssetDatabase 等功能，让生成过程更加健壮，并自动处理路径转换。

结语

通过这个案例，我们看到了代码生成如何化“重复机械劳动”为“一键完成”。CodeGenerator 框架以极低的学习成本提供了清晰的扩展点，而 FilterUtility 则展示了一个真实可用的编辑器工具生成方案。将这一模式推广到项目中的其他类似需求，可以显著降低维护成本，提升开发幸福感。

如果你有兴趣，还可以为框架扩展异步支持或者引入 Roslyn Source Generator，让生成过程在编译时自动触发，进一步与 IDE 体验融合。
