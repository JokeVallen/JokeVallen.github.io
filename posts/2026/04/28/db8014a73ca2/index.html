前言

路径转换是项目开发中常见需求之一，经常会涉及资源路由等应用情景，而这就涉及两个维度，一个是可配置路径的包容性，另一个是路径的合法性，前者包括相对路径和绝对路径，后者则是将输入路径尝试转换为规范路径。本文将以 Unity 项目中以 Assets 为根路径的相对路径转换为例，并附上严格的单元测试代码、测试结果和测试分析。测试代码主要由 AI 根据路径转换部分核心代码进行设计和编写，最终测试结果分析也由 AI 根据测试结果撰写。

代码

  • 统一使用 / 作为分隔符。
  • 所有路径都基于 Unity 项目根目录 Application.dataPath 的父目录进行规范化。

测试覆盖的需求场景

我们围绕实际使用中可能遇到的所有路径输入情况，设计了 7 大类、30+ 个单元测试用例，完整覆盖以下需求场景：

1. 基础合法路径

  • 空/空白输入：null、""、" " → 抛出明确异常
  • 纯 Assets 文件夹："Assets"、"Assets/"、"./Assets" → 统一返回 "Assets"
  • Assets 下的文件/子文件夹："Assets/Scripts/MyScript.cs" → 原样返回（无尾部斜杠）
  • 带中文字符/空格/特殊符号的路径："Assets/我的文件夹/图片.png"、"Assets/My#File%2.asset" → 正确保留

2. 绝对路径转换（跨平台）

  • Windows 绝对路径（正斜杠）："C:/MyProject/Assets/Scripts/test.cs" → "Assets/Scripts/test.cs"
  • Windows 绝对路径（反斜杠）："C:\\MyProject\\Assets\\Prefabs\\Cube.prefab" → "Assets/Prefabs/Cube.prefab"
  • Unix 绝对路径（Linux/macOS）："/home/user/Project/Assets/file.txt" → "Assets/file.txt"（平台条件测试）

3. 路径规范化（解析 . 和 ..）

  • 当前目录 .："Assets/./Subfolder/." → "Assets/Subfolder"
  • 父目录 ..（仍留在 Assets 内）："Assets/Folder1/Folder2/../../Folder3/file.txt" → "Assets/Folder3/file.txt"
  • 冗余斜杠："Assets//Scripts///MyScript.cs" → "Assets/Scripts/MyScript.cs"
  • 尾部斜杠统一去除："Assets/Scripts/"、"Assets/" 均返回无尾部斜杠的路径

4. 相对路径（基于 Unity 项目根目录）

  • 以 ./ 开头："./Assets/Textures/icon.png" → "Assets/Textures/icon.png"
  • 以 ../ 开头（但不逃逸项目根）："../MyProject/Assets/file.txt" → "Assets/file.txt"（假设项目名为 MyProject）

5. 安全防护（目录遍历攻击）

  • 试图逃逸到项目根以外："../../../../Windows/System32/config" → 抛出异常
  • 试图跳转到其他盘符："D:/OtherProject/Assets/file.txt" → 抛出异常
  • 解析后恰好指向项目根目录："Assets/.."、"." → 抛出异常（因为未指向 Assets 文件夹）

6. 非法路径拒绝

  • 大小写错误："assets/icon.png"、"ASSETS/icon.png" → 抛出异常（Unity 资产路径区分大小写）
  • 指向项目根下非 Assets 文件夹："Scripts/MyScript.cs"、"ProjectSettings/ProjectVersion.txt" → 抛出异常
  • 包含无效字符（如 \0）：由 Path.GetFullPath 抛出相应异常
  • 路径过长（超过系统限制）：抛出 PathTooLongException

7. 边界与稳定性

  • 尾部空格："Assets " → 系统自动修剪为 "Assets"，正常返回（满足文件系统行为）
  • 超长路径：已测试并正确抛出 PathTooLongException
  • 跨平台一致性：在 Windows 和 Unix 模拟环境下均按预期工作

2.2 关键结论

评估维度 结论
功能正确性 所有合法输入均被正确转换为 Assets/... 或 "Assets"，且路径中的 .、..、冗余斜杠被规范化。
安全性 成功阻止了所有目录遍历攻击（.. 逃逸、跨盘符、指向项目根目录等）。
容错性 对空输入、大小写错误、非法文件夹均抛出清晰的异常，便于调用方处理。
跨平台 统一使用 / 分隔符，基于 Application.dataPath 动态获取项目根，在 Windows/macOS/Linux 下行为一致。
性能 单次转换仅调用一次 Path.Combine 和 Path.GetFullPath，满足高频调用场景（如 Asset 导入回调）。

2.3 已知限制（文档化）

  • 大小写敏感："assets/" 会被拒绝，因为 Unity 资产路径标准是 "Assets/"。
  • 不会检查文件是否真实存在：只做路径格式转换，不访问磁盘。

总结

经过 30+ 个单元测试的全面验证，ConvertToAssetPathInternal 方法具有以下特性：

  • 稳定可靠：覆盖所有常见及边缘路径场景，无未捕获异常。
  • 安全：自动防御目录遍历攻击，严格限定路径在项目 Assets/ 目录内。
  • 易用：输入格式灵活（相对、绝对、带 ./..），输出统一为 Assets/...。
  • 跨平台：基于 Unity API 和 .NET 标准路径处理，行为一致。

适用于：Unity 编辑器工具开发（如自定义 Asset 引用、配置表导入、资源校验等需要将用户输入路径转换为资产路径的场景）。

不适用于：运行时动态路径转换（应使用 Resources 或 Addressables 体系）。
