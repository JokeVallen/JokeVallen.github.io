AssetBundle（二）

更新于2026-04-12|Unity开发
|浏览量:

阅读本系列文章前建议先阅读官方文档，本系列文章将尽可能避免赘述官方文档，而是偏向于对官方文档的解释和补充。

内容由博主和 AI 协助创作，经由博主审阅，读者仍需审慎阅读。

设计与分组

概述：确定哪些资源放入 AssetBundle，以及如何组织 Bundle 的粒度与依赖关系。

资源清单梳理

  • 列出项目中所有需要动态加载的资源（场景、预制体、纹理、音频、模型等）。

  • 识别共享资源（如公共材质、Shader、全局音频），避免被重复打包。

分组策略选择

常用三种策略（可混合）：

  • 逻辑实体分组：按功能模块（角色、UI、关卡）打包，便于 DLC 和热更新。

  • 类型分组：按资源类型（音频、本地化文本）打包，利于平台复用。

  • 并发内容分组：将同一场景或同一时刻需要加载的资源打包在一起，提升加载速度。

将共享资源单独提取为“共享 Bundle”，避免冗余。

依赖关系设计

  • 确保依赖关系无循环（A 依赖 B，B 又依赖 A）。
  • 规划好 Bundle 之间的依赖树，决定哪些 Bundle 会被其他 Bundle 共同依赖。

注意事项

  • 避免将经常变动的资源与稳定的资源放在同一个 Bundle，否则更新成本高。
  • 单个 Bundle 大小建议控制在 5~20 MB（移动端），过大会增加加载失败风险和解压内存峰值。

构建与部署

概述：将资源打包成 AssetBundle，并部署到服务器或打包进安装包。

构建

  • 脚本构建：通过官方 API 调用构建函数，该方式能够提供高度定制化的构建支持。
  • 可视化工具构建：诸如 AssetBundle Browser、Addressables、YooAsset 等。

检查构建结果

  • 包含 AssetBundleManifest 的 Bundle：构建后会在输出目录生成一个与目录同名的 Bundle（如 AssetBundles），内含 AssetBundleManifest 对象，用于查询依赖。
  • 其它 Bundle：与构建前的设计与分组清单进行对照，避免遗漏或者资源冗余。
  • .manifest 文件：这些用于开发阶段的调试文件要避免发布到生产环境中。
  • 其它文件：自定义构建可能输出的其它文件，例如版本信息、变更记录等文件。

部署

  • 将 AssetBundle 文件及其它相关文件上传至 CDN 或游戏服务器。
  • 配置正确的 MIME 类型（如 application/octet-stream）以确保下载正常。

注意事项

  • 不同平台（iOS/Android/PC）需分开构建，因为纹理压缩格式、平台特定资源不同。
  • 构建时注意处理资源的 “Asset Bundle Name” 设置（可通过 AssetImporter 或脚本批量设置）。

下载与加载

概述：在客户端运行时，按需下载或加载 AssetBundle，同时处理依赖关系。

版本比对

  • 从服务器获取最新版本清单，与本地缓存清单比对，得到需要下载的 Bundle 列表（新增、更新）。

下载 AssetBundle

  • 使用 UnityWebRequestAssetBundle 进行下载，支持断点续传、超时重试。
  • 优先下载依赖 Bundle，再下载目标 Bundle，或并发下载。

加载 AssetBundle

  • 调用 AssetBundle.LoadFromFile（本地缓存）或 DownloadHandlerAssetBundle.GetContent（下载完成直接获取）得到 AssetBundle 对象。
  • 加载资源：bundle.LoadAsset(“MyPrefab”) 或异步版本 LoadAssetAsync。
  • 实例化资源：Instantiate(prefab)。

依赖处理

  • 通过 AssetBundleManifest 获取某个 Bundle 的所有依赖（GetAllDependencies）。
  • 递归加载所有尚未加载的依赖 Bundle，确保资源引用完整。

注意事项

  • 异步加载时注意在回调中判断加载是否成功，处理失败重试。
  • 避免主线程阻塞，尽量使用异步 API（LoadAssetAsync、LoadFromFileAsync）。
  • 加载完成后可立即卸载原始的 UnityWebRequest 资源，保留 AssetBundle 对象即可。

缓存与更新

概述：减少重复下载，支持热更新。

本地缓存机制

  • 将下载的 AssetBundle 存入应用沙盒（如 Application.persistentDataPath）。
  • 维护一个本地清单，记录已缓存 Bundle 的 Hash 值和文件路径。

增量更新

  • 每次启动对比服务器清单，仅下载 Hash 变化的 Bundle。
  • 对于大版本更新，可直接下载完整的资源包替换旧版本。

回滚与容错

  • 如果更新过程中断，保留旧版本 Bundle，下次启动可继续。
  • 提供“校验”机制，下载完成后校验 Hash，失败则重试或提示用户。

注意事项

  • Unity 的缓存系统（Caching）已部分过时，推荐自行管理文件存储，以获得更灵活的控制。
  • 对于 Steam、App Store 等平台，大版本更新可能仍需通过商店分发，AssetBundle 更适合小版本热更。

卸载与内存管理

概述：及时释放不再使用的 AssetBundle 及其资源，防止内存泄漏。

引用计数

  • 为每个 Bundle 维护加载次数，当计数为 0 时调用 Unload(true) 彻底释放。

场景切换时清理

  • 卸载不再使用的 Bundle，调用 Resources.UnloadUnusedAssets 释放内存。

实例与资源的生命周期

  • 从 Bundle 中实例化的 GameObject 在销毁时，不影响 Bundle 本身。
  • Bundle 必须在所有从中加载的资源都销毁后才能安全卸载。

注意事项

  • 切勿忘记卸载 Bundle，否则内存会持续增长。
  • 使用 Unload(false) 仅卸载 Bundle 本身，已加载的资源仍保留，但无法再从该 Bundle 加载新资源，容易造成孤立资源。除非有特殊需求，否则一般用 Unload(true)。
  • 对于频繁加载卸载的场景（如关卡切换），注意观察 Profiler 确保内存无泄漏。

流程建议

该部分内容由 AI 总结，经由博主审核纳入本文，请审慎阅读。

设计与分组

分组粒度：避免“一个资源一个 Bundle”与“一个大包”两个极端

  • 过细的粒度（每个资源独立 Bundle）：

    • 优点：更新粒度小，下载量少。
    • 缺点：大量小文件导致加载请求过多，依赖关系复杂，磁盘碎片化，总体积可能因冗余而膨胀（Unity 对每个 Bundle 有固定开销）。

  • 过粗的粒度（一个巨型 Bundle）：

    • 优点：加载管理简单，依赖单一。
    • 缺点：更新时需整体下载，内存加载峰值高，灵活性差。

  • 推荐做法：

    • 按功能模块（UI、角色、场景）打包，每个 Bundle 大小控制在 5~20 MB（移动端）或更大（PC/主机）。
    • 对于共享资源（Shader、公共材质、字体），单独提取为“共享 Bundle”，被其他模块依赖。

场景 Bundle 的“自包含”原则

  • 将场景及其独有资源打包在一起，避免场景加载时再去动态拉取其他分散的 Bundle，减少加载时的依赖等待。
  • 如果某些资源（如全局音效）被多个场景共享，则保留在独立的“共享 Bundle”中，场景加载前确保共享 Bundle 已预先加载并常驻内存。

变体功能的慎用与替代

Unity 的 AssetBundle 变体（Variants）可以在打包时切换资源（如高清/标清纹理），但在实际项目中容易因依赖关系复杂而出错。

替代方案：

  • 通过 Addressable 的“资源组 + 标签”实现不同画质资源包。
  • 手动在运行时根据设备等级加载不同精度的资源，而非依赖变体系统。

动态分析依赖关系

使用 AssetBundle Browser 或 自定义脚本 分析构建后的依赖图，识别：

  • 循环依赖（需重构）。
  • 高冗余资源（如某纹理被 10 个 Bundle 各自包含，应提取到共享 Bundle）。
  • 无用的依赖（Bundle 引用了实际未使用的资源）。

构建与部署

构建时启用确定性构建

  • 设置 BuildAssetBundleOptions.DeterministicAssetBundle，确保相同资源每次构建的 Bundle 文件内容一致（减少更新时因内容变化导致的错误下载）。

利用增量构建加速迭代

  • 使用 BuildAssetBundleOptions.AppendHashToAssetBundleName 或手动维护资源 Hash，仅重新构建变更的 Bundle，而非全量打包。
  • 在 CI/CD 中，通过比较文件修改时间或 Git 变更列表，只打包受影响的部分。

压缩算法的选择

  • LZ4：解压速度快，适合运行时按需加载（默认采用 ChunkBasedCompression）。
  • LZMA：压缩率高，但解压极慢，适合作为下载包压缩（在服务器端以 LZMA 存储，客户端下载后解压为 LZ4 再加载）。
  • 不压缩：适用于已压缩的资源（如视频、加密包）或内存映射加载的场景。

平台特定处理

  • 纹理格式：移动端建议 ASTC（Android/iOS），PC 建议 DXT/BC。不同平台需分别构建。
  • 音频压缩：根据平台选择 Vorbis（PC/Mac）、ADPCM（移动端）、MP3 等。
  • 使用 BuildTarget 明确指定平台，构建目录按平台隔离（如 Build/Android/、Build/iOS/）。

部署时的版本清单设计

  • 服务器提供 version.json，包含每个 Bundle 的名称、Hash、大小、依赖列表。
  • 采用差异更新：客户端携带本地 Hash 列表，服务器返回需要更新的 Bundle 列表。
  • 对于大版本更新，可直接提供“版本包”下载，客户端直接替换整个资源目录。

下载与加载

下载策略

  • 并发下载：同时下载多个 Bundle，但需控制并发数（建议 2~4 个），避免耗尽连接数。
  • 优先级队列：关键资源（如启动场景）优先下载，非紧急资源（如商城皮肤）后台下载。
  • 断点续传：UnityWebRequest 本身不支持断点续传，需自行实现 Range 请求，或使用成熟下载库（如 BestHTTP）。

加载顺序与依赖预加载

  • 加载目标 Bundle 前，先通过 AssetBundleManifest.GetAllDependencies 获取所有依赖，递归加载尚未缓存的依赖。
  • 可对依赖树进行扁平化排序，确保无循环依赖。
  • 将加载操作放在协程或 async/await 中，避免主线程阻塞。

异步加载的性能考量

  • LoadAssetAsync 会在后台线程解压/解析数据，但回调仍在主线程执行。大量同时异步加载可能导致主线程回调过载，可分批或使用对象池管理回调。
  • 对于大量小资源（如 UI 图标），考虑将它们合并到一个 Bundle 中，减少异步操作数量。

避免重复加载同一Bundle

  • 维护已加载 Bundle 的引用计数或字典，相同 Bundle 不重复加载。

注意：AssetBundle.LoadFromFile 如果重复调用同一路径，会返回新的 AssetBundle 对象，但实际文件会被映射两次，内存中可能存在重复数据。最好通过缓存保证每个 Bundle 只加载一次。

WebGL 平台特殊处理

WebGL 不支持多线程，也不支持 UnityWebRequest 的缓存系统（Caching 无效）。

解决方法：

  • 将 AssetBundle 放在 StreamingAssets 文件夹内（随安装包）。
  • 或使用 IndexedDB 通过 JavaScript 插件实现本地缓存，在 C# 侧通过 UnityWebRequest 加载 URL，并配合 File.WriteAllBytes 手动管理。

注意：WebGL 内存限制，加载大 Bundle 可能导致崩溃，需分包更细。

缓存与更新

本地缓存目录结构

  • 使用 Hash 或版本号作为文件名，避免覆盖冲突。
  • 下载完成后原子性移动到正式目录（先下载到临时文件，再重命名）。

版本更新的原子性

  • 更新过程中如果程序崩溃，下次启动时应能恢复：
  • 保留旧版本清单，新 Bundle 下载完成后先验证 Hash，再更新清单文件。
  • 清单文件更新可以采用“先写入临时文件，再替换”的方式保证原子性。

资源回滚与降级

  • 服务器保留多个版本资源，客户端可强制回退至稳定版本（如发现新版本严重问题）。
  • 客户端加载失败时，可尝试降级到较低质量资源或显示默认占位资源，避免功能不可用。

卸载与内存管理

引用计数机制设计

  • 为每个 Bundle 维护一个 RefCount，每次加载时 ++，卸载时 –。
  • 当计数归零时，调用 Unload(true) 并移除缓存。

注意：从 Bundle 中 Load 出的资源（如 Prefab）如果被实例化，其生命周期独立，但 Bundle 必须保持加载直到所有实例及其依赖被销毁。可以使用弱引用或自定义资源管理器统一追踪。

场景切换时的自动卸载

  • 在场景切换时，卸载所有非持久化（不常驻内存）的 Bundle。
  • 调用 Resources.UnloadUnusedAssets 释放未引用的资源。

注意：UnloadUnusedAssets 是异步操作，且比较耗时，应在合适时机（如 loading 界面）调用。

纹理与音频的流式加载优化

  • 对于超大纹理，使用 Texture Streaming（Unity 2018+）功能，让纹理以 mipmap 级别流式加载，避免一次性加载全部。
  • 对于长音频，使用 AudioClip 的流式加载（LoadType = Streaming），边播边加载。

内存泄漏排查工具

  • 使用 Unity Profiler 的 Memory 模块，查看 AssetBundle 对象数量和驻留资源。
  • 利用 Memory Profiler 包（Package Manager）深度分析哪些资源未被卸载。
  • 在关键场景（如关卡结束）前做一次 UnloadUnusedAssets 并观察内存曲线。

团队协作与工具链

使用成熟的资源管理插件

Addressable 在 AssetBundle 之上提供了更友好的资源管理 API，自动处理依赖、引用计数、异步加载和更新。它的“资源组”可配置打包策略（Pack Together / Pack Separately / Pack Together By Label），并支持远程更新与本地缓存的整合。如果项目尚在早期，强烈建议直接使用 Addressable，而非裸用 AssetBundle。

构建自动化与 CI/CD 集成

  • 编写构建脚本时，支持命令行调用（-executeMethod），便于集成到 Jenkins、GitLab CI 等。
  • 构建产物（Bundle + 清单）按版本号归档，方便回滚。
  • 自动上传至 CDN，并生成更新通知。

调试与监控

  • 开发环境下可开启 BuildAssetBundleOptions.ForceRebuildAssetBundle 确保完整性。
  • 使用 AssetBundle.LoadFromFile 时，可打印加载时间和内存增量，记录性能基准。
  • 在生产环境收集 Bundle 下载成功率、加载耗时等数据，便于优化。

特别说明

很多成熟的插件针对上述流程进行了简化，提供更人性化和现代化的资源管理方式，例如 Addressables、YooAsset 等。优先建议使用这些插件，除非它们无法提供你需要的服务，或者你需要更加定制化的资源管理方式，亦或者用于学习和尝试。
